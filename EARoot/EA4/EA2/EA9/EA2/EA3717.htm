<font color="#e0121d"><b>Meta-Modeling – Existing Support for Subclasses and Attributes</b></font><br/><font color="#29313b">The dynamic addition of attributes and subclasses is already supported in the SID via two patterns, EntitySpecification/Entity and the CharacteristicSpecification/CharacteristicValue, sometimes referred to as the Dynamic Object pattern. They will be briefly revisited in the section Error: Reference source not found in this part of the guide book, where inheriting attributes (and associations) will be described.</font><br/><font color="#29313b">The use of these two SID patterns is also often referred to as dynamic sub-classing.  But what does this mean? It means that a subclass of the applicable EntitySpecification and/or a subclass of the applicable Entity are implied, rather than explicitly modeled. The and/or is based on the instances of CharacteristicSpecifications related to the instance of the EntitySpecification.</font><br/><font color="#29313b">If all the CharacteristicSpecValue instances are fixed (cannot be configured), then there is only an implied subclass of the EntitySpecification. If all the CharacteristicSpecValue instances are configurable, then there is only an implied subclass of the Entity. If the CharacteristicSpecValue instances are fixed and configurable, then there are implied subclasses of the EntitySpecification and the Entity.</font><br/><font color="#29313b">An application could provide the capability (user interface and logic) to add instances of CharacteristicValue when adding an instance of a RootEntity. This capability would determine if the RootEntityType that describes the RootEntity has any related instances of CharacteristicSpecification. The dynamicAttrAssoc attribute of RootEntityType would be used to determine this.</font><br/><font color="#29313b">Implementing the meta-model to support the dynamic addition of classes and subclasses, both implicit and explicit, means that inheriting attributes must be supported. Existing SID entities should be handled by transformation of the information model to the data model by rolling down the super-classes. A representative set of application logic that supports inheritance for dynamically added classes and subclasses is described next.</font><br/><font color="#29313b">Application logic can be employed to support inheriting attributes by copying instances of CharacteristicSpecifications and CharacteristicSpecValues to subclasses for dynamically added classes and subclasses, both implicit and explicit. The application would create instances of RootEntityTypeCharUse and RootEntityTypeCharValueUse. If copying, rather than true inheritance, is done, then the ability to change the properties (attributes) of the Characteristic entities of the subclasses should be prevented by the application to avoid inconsistencies. The application could also ask if selected instances of RootEntityTypeCharValueUse should also be created.</font><br/><font color="#29313b">For those SID implementer’s who decide to use a meta-model like the one described here, there is an important point to keep in mind. Existing associations between entities that represent specifications and applications of the characteristics or directly to characteristics are no longer necessary. The section Error: Reference source not found in part of the guide book will further explain this.</font><br/><font color="#29313b"><br/></font><font color="#e0121d"><b>Meta-Modeling Associations</b></font><br/><font color="#29313b">Requirements for meta-modeling associations, like those for entities and attributes, can be gathered by looking at the properties of an association and its roles in a UML tool.
