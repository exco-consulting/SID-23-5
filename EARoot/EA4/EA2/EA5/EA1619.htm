<font color="#e0121d"><i>Resource Domain Executive Summary</i></font><br/><font color="#29313b">Part of this document describes Computing and Software Resources within the Resource domain of the Information Framework (SID).  Included is support for digital services, such as Workplace as a Service (WpaaS), multi-tenancy offerings, support for virtualization and network resource abstraction by the introduction of the Resource Function concept.</font><br/><font color="#29313b">The move to virtualization, hybrid networks using both traditional Network Appliances and Network Applications software running on virtual machines or container technologies introduces a much more dynamic relationship between the network capabilities and how they are realized at any instance in time by resources.</font><br/><font color="#29313b">The introduction of the Concept of Network as a Service promoted in TMF909 NaaS API Component Suite [1] and Open Digital Architecture (ODA) GB999 ODA Production Implementation Guidelines [2] - introduces the notion of abstracting network capabilities by separating the ‘What‘ is provided, from ‘How‘ it is provided. This allows uniform network services to be defined independent of specific technologies and their realization which may not be known at design time. Example of where this is useful for 5G/4G based services where the switch from 4G to 5G access needs to be transparent to the consumer of the Network Service. </font><br/><font color="#29313b">Evolution of Network implementations from network appliances to network applications as software running on compute infrastructure requires adjustment to the traditional resource management models which reflect static assets that take time to deploy and configure, and which change infrequently to a model that can handle dynamic run time change in these relationships such as encountered with Network Function Virtualization (NFV).</font><br/><font color="#29313b">Computing Resources, while not explicitly modelled, are supported by a number of SID Aggregate Business Entities (ABEs) including Physical Resource Specification, Logical Resource Specification, Compound Resource Specification, and their related “entity” ABEs, such as Physical Resource.</font><br/><font color="#29313b">Software Resources are explicitly modelled within the SID and supported by a number of current ABEs, including Logical Resource Specification, Software, and Operating System.</font><br/><font color="#29313b">The Compound Resource Specification, Compound Resource, Software ABEs have been enhanced.  The Operating System ABE is now supported by the Software Specification and Software ABEs and has therefore been removed.</font><br/><font color="#29313b">Support for a number of scenarios, which includes, but is not limited to, Software compatibility and WpaaS, and vCPE are also contained within this document.</font><br/><font color="#e0121d">Resource Domain Overviews</font><br/>The Resource domain represents roles, information and activities carried out by parties (e.g. individuals / organizations) playing roles that are involved in the strategic planning, definition, development and operational aspects of Resources (e.g. functions, applications, computing, networking and storage) that represent the infrastructure of an enterprise that are used to realize Services. Activities include management of: strategies, capabilities, lifecycles, catalogs, inventories, topologies, installations, activations, alarms, problems, performance, mediation, usage statistics and support of Resources that are managed by an enterprise.<br/><br/>The Resource Domain provides a view of the resource portion of the information model, which includes mined information from other information models. The SID draws from the work of ITU-T, IETF, ONF, MEF and DMTF. This model is built with the express purpose of serving as a foundation for other Resource (sub-) models to fit into.<br/>
